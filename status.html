改善ログ ― 何を / どんな改善 / なぜ至ったか

14周の記録（各周＝1つの穴を塞ぐ）

①〜⑧ 既知監査の穴を解消 ⑨〜⑭ レビューで新発見した論点
①NG表現の正規化バイパス封じ柔整/あはき・薬機・景表
何を
広告コピーのNGワードチェッカー（checkCopy）
どんな改善
正規化 normalizeForCheck（NFKC＋全空白除去＋大文字化）を通してから照合。辞書側も同じ正規化。
なぜ
実際に叩いたら 「Ｎｏ．１」「１００％」「絶　対」「な お る」 が素通りしていた。全角・文字間スペース・ひらがな開きで規制を回避でき、業務停止/罰金の芽になる。
②症状名×効能想起の共起ブロック柔整/あはき（適応症標榜）
何を
要配慮の"推知"検出（detectSensitiveInference）
どんな改善
「症状名＋効能を想起させる語」の共起を検出し、接骨院/鍼灸（enumerated業種）ではhighに格上げしてブロック。
なぜ
「肩こりがスッキリ軽くなります」がpassしていた。症状名の単独言及は可でも、効能想起と組むと"適応症の標榜"に近づく。業種で厳格度を変える必要があった。
③英語/ローマ字＋カタカナ開き薬機・景表
何を
NG辞書の言語カバレッジ
どんな改善
英語（cure/guaranteed/the best 等）とカタカナ変種（ゼッタイ/ナオリ）を追加。一般語と衝突する heal/best は誤検知回避でmedium。
なぜ
「Guaranteed to cure」「the BEST」「ゼッタイに良くなる」が素通り。日本語辞書だけでは英語混じり・カタカナ開きの誇大表現を止められなかった。
④要配慮ガードの再帰対応APPI 要配慮個人情報
何を
症状/診断の非結合ガード（stripSensitive）
どんな改善
大文字小文字を無視＋ネスト/配列を再帰走査して剥離。剥がしたキーパスを記録。
なぜ
{ Symptom（大文字）, profile.diagnosis（ネスト） } が剥がれず、要配慮情報が実名に結合され得た。トップ階層・小文字キーしか見ていなかった。
⑤保持期間 purgeAPPI 安全管理措置
何を
計測データの保存期間管理（purgeExpired / POST purge）
どんな改善
最終アクセスから既定730日を過ぎたセッション/box_stats/identity/tag_firesをカスケード消去（cron想定）。
なぜ
schema.sqlに「古い行をpurge」の方針はあるがコード実装が無く"設計止まり"だった。保存期間の管理は安全管理措置の要。実行できる形に落とした。
⑥外部送信規律の通知改正電気通信事業法
何を
外部送信の通知（buildDisclosure / GET disclosure）
どんな改善
「何を・どこへ・何のため」を開示。第三者提供なし・要配慮除外を明示。ポリシーURLは店舗が設定（代筆しない）。
なぜ
コード内に通知の仕組みが1つも無かった（grep 0件）。SDKが閲覧履歴等を送る以上、送信情報・送信先・目的の通知/公表義務が残る。
⑦CSV持ち出しの安全管理APPI 安全管理措置
何を
開示請求向けCSVエクスポート（POST export）
どんな改善
持ち出しログ（actor必須）＋CSV数式インジェクション対策＋任意AES-256-GCM暗号化（scrypt鍵導出）。
なぜ
監査がCSV持ち出しの漏えいリスクを挙げていたがエクスポート機能自体が未実装だった。作るなら最初から安全に。=csvは表計算での数式実行リスクもあるため無害化。
⑧Google連携のスコープ最小化Google APIs 規約
何を
Search Console連携（assertReadonlyScope）
どんな改善
必要スコープを webmasters.readonly に固定し、書込/サイト認証scopeが混じれば実行時に拒否。
なぜ
fetchSearchConsoleがスコープを強制しておらず、過剰権限のトークンでも動いてしまう状態だった。スコープ最小化・目的外利用禁止の担保。
⑨RLSテナント分離（アプリ層）APPI 安全管理措置
何を
テナント間のデータ隔離（tenantOfFriend / x-tenant-id）
どんな改善
2テナントのseedを用意し、自テナント外のjourney/tags/export/disclosureを403。DB側はschema.sqlのRLSポリシーと二重で担保。
なぜ
RLSがlegal-audit/schema.sqlで"設計"止まりで、デモは単一テナント＝アプリ層に分離が無かった。他店のデータが混ざる事故を防ぐため実装＋テストへ格上げ。
⑩個人関連情報の第三者提供APPI 第31条
何を
結合(merge)時の同意 確認・記録（consent_record）
どんな改善
同意の由来（誰が・どの方法で・いつ取得したか）を記録。揃わない同意は"不完全"としてフラグ、証跡APIで引ける。
なぜ
網羅性レビューで発見。Loku側では匿名視線＝「個人関連情報」。店舗が実名結合＝提供先で個人データ化する構図で、提供元(Loku)に本人同意の確認・記録義務が生じる。同意フラグを受けるだけでは足りなかった。
⑪越境移転／委託先の監督APPI 第28/25条
何を
委託先レジストリ＋越境判定（subprocessors）
どんな改善
委託先（Supabase/Google/LINE）と処理目的・保管国を管理し越境を判定。保管国"未設定"は"要確認"に倒す（誤って国内扱いしない）。保管国は運用者が登録＝当方で断定しない。
なぜ
レビューで発見。データが国外保管なら移転先国等の情報提供義務(28条)＋委託先監督(25条)。保管国を把握していないと越境の有無すら判定できない。事実に反する国名は書けないので"欄"にした。
⑫プロファイリングの透明性・拒否権利用目的の特定／消費者保護
何を
タグ付けロジックの開示と停止（profiling-info / opt-out）
どんな改善
どのタグを・どの注視・どの閾値で付けるかを開示。拒否すると既存タグを撤回＋以後は再付与しない。
なぜ
レビューで発見。「ホットリード」等のタグ付け＝見込み度合いの自動プロファイリング。本人が基準を知れず断れないのは不透明で、消費者の納得感・利用目的特定の観点で望ましくない。
⑬未成年の同意（保護者同意）APPI 同意能力
何を
未成年フラグ＋保護者同意の記録（is_minor / guardian_consent）
どんな改善
未成年は「本人＋保護者同意」が揃って初めて記録完全。保護者同意なしは不完全フラグ。
なぜ
レビューで発見。施術所の患者には未成年が含まれ得て、症状（要配慮）も絡む。低年齢本人の同意は有効と認められにくく、法定代理人の同意が要る。
⑭開示・削除請求への本人確認APPI 開示・利用停止等
何を
本人起点の請求の本人確認（subject_request / identity_verified）
どんな改善
本人起点(subject_request)の開示/削除はidentity_verified必須、未了なら403。確認の有無を証跡に記録。社内運用の請求とは区別。
なぜ
レビューで発見。export/forgetがactor(操作スタッフ)しか見ておらず、請求者＝本人の担保が無い。なりすましで他人のデータを開示・不正削除される恐れがあった。
次の一手 ― まだ発掘余地のある論点

ここから先に回せる候補

  • ⑮ 目的外利用の防止／利用目的の変更・再同意 ― 目的を後から広げる時に再同意を要求する仕組み
  • ⑯ 漏えい時の報告・本人通知（APPI26条）への備え ― インシデント記録と通知フローの型
  • ⑰ Cookie/匿名IDの取得タイミング ― 取得前通知/同意の精緻化（現行は取得＋通知）
  • ⑱ 保有個人データに関する事項の公表 ― 利用目的・請求手続きの常時開示ページの型
