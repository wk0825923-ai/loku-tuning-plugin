法務監査 / LEGAL AUDIT

Loku Tuning 導入の法務リスク監査

「業務停止に抵触しないか」「個人情報の取扱い」を中心に、日本法で確認した論点と対策。

⚠ 本資料は社内検討用の整理であり、法的助言ではありません。実運用前に、個人情報・広告規制・電気通信事業法に詳しい弁護士／個人情報保護委員会窓口での確認を推奨します。出典は各項末尾。

結論（TL;DR）

設計次第で「グリーン」。ただし2点を外すと業務停止・行政処分の芽がある。

① 業務停止の直接リスクは「施術所の広告規制」（接骨院=柔整法／鍼灸=あはき法）。広告できる事項が法定列挙で、違反は30万円以下の罰金＋一定期間の業務停止・免許取消。だからコピーの自動生成・自動公開はしない（＝当初からの方針が正しい）。
② 個人情報の直接リスクは「要配慮個人情報」。整体/接骨院の症状（肩こり・腰痛）と個人を結びつけると要配慮個人情報になり、取得に事前同意が必須・オプトアウト第三者提供は不可。→ 症状そのものは結合せず行動データのみ・明示同意で結合する設計にする。

STATUS / 解消ステータス

どこまで"解消"したか（3層で管理）

「法的に100%セーフ」は弁護士でも言い切らない性質のもの。機械で防げる部分はコード＋テストで解消し、残りは設計・運用・専門家で管理する。＝"解消できない"のではなく、機械で解消できる分は解消済み。

層対象状態
機械で解消済
コード＋テスト		NG表現ブロック(正規化で全角/空白/ひらがな/英語/カタカナ回避も遮断)／症状×効能の共起ブロック／症状の非結合(再帰strip)／同意ゲート／削除(forget)／配信停止(opt-out)／健康語の推知フラグ／保持期間purge／外部送信通知／CSV安全管理(持ち出しログ+暗号化)／Google readonly強制／監査ログテストで担保
設計で低減RLSテナント分離／アクセス制御（DBはschema.sqlのRLS・アプリ層もx-tenant-idで403＝実装+テスト済）DB+アプリ層で担保
運用・専門家
機械では解消不可		DPA委託契約／従業者教育／特商法の個別表記／弁護士・個情委の最終確認運用で担保

最下段だけは性質上ソフトウェアで解消不可（人が担う領域）。それ以外は解消し続ける。

RISK MAP

リスク一覧（重大度順）

重大
① 施術所の広告規制 ＝ 業務停止の直接原因
柔道整復師法24・30条／あはき法／あはき・柔整広告ガイドライン（令和7年2月18日改定）
接骨院・整骨院（柔整）／鍼灸・あん摩（あはき）は広告可能事項が法律で列挙され、それ以外は原則NG。「治る」「効果」「No.1」「体験談」等は違反方向。違反は30万円以下の罰金、さらに一定期間の業務停止・免許取消が命じられ得る。
誰が処分される？ → 施術所（＝Lokuの店舗ユーザー）。ただしLP文言を当方が自動生成・公開して違反を作れば、加担・信用毀損リスク。
対策
  • コピーの自動生成・自動公開はしない（構造・導線の提案まで／文言は必ず人が承認）
  • NGワードチェッカーを必須化（柔整・あはき／薬機法／景表法の禁止表現を出稿前にブロック）
  • 業種フラグで出し分け：無資格の「整体・リラク」と有資格の「接骨院・鍼灸」で規制が異なる。テナント種別で許容表現を切替
  • 広告可能事項テンプレを用意し、逸脱は下書き止まり＋警告
出典: 厚労省 あはき・柔整広告ガイドライン概要（令和7年）／柔道整復師法
重大
② 要配慮個人情報（健康・症状）の結合
個人情報保護法（APPI）／個人情報保護委員会ガイドライン
接骨院等は個人情報保護法の対象。傷病・症状などの健康情報＋個人＝要配慮個人情報。取得は本人の事前同意が原則必須、オプトアウトによる第三者提供は不可。匿名の視線ログをLINE友だち（実名）に結合した瞬間、それは個人情報になる。
誰が処分される？ → 店舗（個人情報取扱事業者）と、処理を担うLoku。個人情報保護委員会の指導・勧告・命令（命令違反に罰則）。
対策
  • 症状そのものは結合しない：結合するのは「どのボックスを何秒見たか」の行動のみ。要配慮情報（診断・症状申告）は結合対象外にする（＝要配慮該当を避ける設計）
  • 明示同意で結合：友だち追加時に「閲覧データを結合して案内に使う」旨のオプトインを取得（同意なしは匿名集計のみ・結合しない＝実装済みの同意ゲート）
  • 利用目的の特定・通知／安全管理措置：プライバシーポリシー明記、テナント分離（RLS）、アクセス制御
  • 開示・削除・オプトアウトへの対応導線（friend単位で結合解除）
  • 検索クエリの健康語（肩こり等）は要配慮の"推知"に配慮（推知フラグを立て、結合同意・表示を強める）
出典: 個人情報保護委員会 要配慮個人情報FAQ
中
③ 外部送信規律（日本版Cookie規制）
改正電気通信事業法（2023年6月16日施行）
Cookie・閲覧履歴・IP・端末情報を外部送信する際、送信情報の内容・送信先・利用目的の通知または公表が必要。サービス（サイト/アプリ）単位で対象、自社集客サイトも対象になり得る。
Tuningは自社1st-party配信で送信先＝Loku自社のため第三者送信は限定的だが、通知・公表義務は残る。
対策
  • 各店LPに外部送信の通知（何を・どこへ・何のため）とプライバシーポリシーへの導線を標準テンプレで自動設置
  • 第三者タグ（広告等）を足す場合は送信先を明示・最小化
出典: 総務省 外部送信規律
中
④ 景表法・薬機法（表現の誇大・効能）
景品表示法／医薬品医療機器等法
「必ず痩せる」「〇〇が治る」等の優良誤認・効能効果の標榜。改善提案でコピーを触るほどリスクが増える。
対策
  • ①のNGワードチェッカーに景表・薬機の辞書も統合／実証データのない効果表現をブロック
低
⑤ プラットフォーム規約（LINE）
LINE公式アカウント／Messaging API 利用規約
友だち情報の利用範囲・配信頻度・オプトアウトは規約順守。過度なリマインドはブロック増・規約抵触。
対策
  • 配信頻度の上限・停止導線を既定化。カゴ落ちリマインドも回数制限

通知・スタッフ招待・Google連携・データ持ち出しを追加したため、再監査した（いずれも裏取り済み・重大度は中〜低）。

中
⑥ LINE配信（通知・カゴ落ちリマインド）
LINE公式アカウント 利用規約／ガイドライン・特定商取引法
過度・ユーザーの反応を無視した配信はアカウント利用停止（BAN）リスク。商品を訴求する配信は特商法の表示義務がかかる場合がある。
対策
  • 配信頻度の上限（実装済）＋お客様がいつでも配信停止できる導線
  • ブロック／既読など反応を見て抑制（送りっぱなしにしない）
  • 商品訴求時は特商法表記を添える
出典: LINE公式アカウントガイドライン
中
⑦ スタッフ招待（従業者が顧客データにアクセス）
個人情報保護法 安全管理措置（人的・組織的）
スタッフが実名×行動データを閲覧＝従業者の監督義務（役割・アクセス権限の設定、就業規則、違反時の懲戒）。
対策
  • 権限の最小化（「見るだけ／送信もOK」の別＝実装済）
  • アクセスログ・秘密保持／就業規則・退職時のアクセス削除
出典: 個人情報保護委員会 通則ガイドライン（安全管理措置）
低
⑧ Google連携（サチコ取込）
Google APIs 利用規約
利用規約への同意・プロパティの読取権限・スコープ最小化・クォータ回避禁止・目的外利用の禁止。
対策
  • readonly スコープに限定・取得データは当該店の分析のみに使用
出典: Search Console API 利用規約
低
⑨ データ持ち出し（CSVエクスポート）
個人情報保護法 安全管理措置
開示・持ち出しは権利対応として妥当だが、書き出したCSVの漏洩リスク。
対策
  • 暗号化／アクセス制御・持ち出しログ・用途明示
中
⑩ 個人関連情報の第三者提供（提供元の確認・記録義務）
個人情報保護法 第31条（個人関連情報）
Loku側では匿名の視線データ＝「個人関連情報」（それ単体では個人データでない）。これを店舗がLINE実名と結合＝提供先で個人データになると想定される場合、提供元（Loku）は本人同意が取得済みかを確認し、記録する義務がある（APPI31条）。同意フラグを受けるだけでは記録義務を満たさない。
誰が処分される？ → 確認・記録を怠った提供元（Loku）。個人情報保護委員会の指導・命令の対象。
対策
  • 結合(merge)時に同意の由来（誰が・どの方法で・いつ取得したか）を記録（consent_record）
  • 記録が揃わない同意は"不完全"としてフラグ化し可視化（consent-record API で証跡を引ける）
  • 本人同意は提供先（店舗）が取得し、提供元（Loku）はその事実を確認・保存
出典: 個人情報保護委員会 個人関連情報ガイドライン（31条）
中
⑪ 外国にある第三者への提供（越境移転）
個人情報保護法 第28条／委託先の監督 第25条
計測データの保管・処理を外部サービス（DB・検索連携・配信基盤）に委託し、データが日本国外に保管される場合、本人への移転先国・当該国の制度・講じる措置の情報提供が必要（28条）。また委託先は監督義務の対象（25条）。委託先の保管国を把握していないと、そもそも越境の有無が判定できない。
誰が処分される？ → 情報提供・監督を怠った提供元（Loku）。
対策
  • 委託先レジストリで処理目的と保管国を管理（subprocessors API）
  • 越境判定：国内=通知不要／国外=28条の情報提供が必要／未設定は"要確認"として通知対象に倒す（誤って国内扱いしない）
  • 保管国は運用者が登録（当方で断定しない＝誤判定回避）
出典: 個人情報保護委員会 外国third提供ガイドライン（28条）

GO-LIVE CHECKLIST

業務停止に抵触しないための導入前チェック

実装済=コード実装＋テスト済 ／ 設計=スキーマ/設計に反映 ／ 運用=導入時の運用・書面

  • 文言の自動公開を無効（構造/導線提案まで・コピーは人承認）実装済 同意ゲート＋公開ブロック
  • NGワードチェッカー（柔整・あはき・薬機・景表）＋業種フラグで出し分け実装済 compliance.mjs / check-copy API・正規化(NFKC＋空白除去＋大小文字)で全角/文字間スペース/ひらがな開きの回避も遮断・症状名×効能想起の共起は接骨院/鍼灸でブロック(適応症標榜)
  • 結合は行動データのみ・症状は非結合（要配慮該当を回避）実装済 stripSensitive()・大文字キー/ネスト/配列内も再帰剥離
  • 友だち追加時の明示同意／非同意は匿名集計のみ実装済 merge consentゲート
  • 個人関連情報の提供＝同意の確認・記録（APPI31条）実装済 merge時にconsent_record(誰が/方法/日時)を記録・不完全はフラグ・consent-record APIで証跡・回帰テスト済
  • 越境移転の情報提供・委託先の把握（APPI28/25条）実装済 subprocessors APIで委託先レジストリ＋越境判定・保管国未設定は"要確認"に倒す・保管国は運用者が登録・回帰テスト済
  • 開示・削除・オプトアウト（friend単位で削除）実装済 forget API
  • プライバシーポリシー・外部送信通知を各店LPに標準設置実装済 訪問者UIに通知＋同意・buildDisclosure()/GET disclosureで「何を/どこへ/何のため」をLP埋め込み(第三者提供なし・要配慮除外を明示・ポリシーURLは店舗設定)
  • 安全管理措置：テナント分離（RLS）・アクセス制御実装済 DB=schema.sqlのRLSポリシー(tenant_id=auth.jwt)／アプリ層もtenantOfFriend+x-tenant-idで自テナント外のjourney/tags/export/disclosureを403(回帰テスト済)／保持期間purge実装済 purgeExpired()/POST purge・最終アクセスから既定730日で計測データを消去(cron想定)・回帰テスト済
  • 委託契約（DPA）：店舗＝取扱事業者、Loku＝委託先を書面化運用
  • 配信頻度の上限＋配信停止（オプトアウト）導線（LINE規約・特商法）実装済 頻度上限＋停止を明示
  • スタッフの権限最小化・退職時アクセス削除（従業者の監督）実装済 見るだけ/送信の別
  • Google連携は readonly スコープに限定（Google APIs規約）実装済 assertReadonlyScope()でwebmasters.readonly以外(書込/サイト認証)を実行時に拒否・fetchSearchConsoleが強制
  • 持ち出しCSVの安全管理（暗号化・アクセス制御・ログ）実装済 POST export・actor必須で持ち出しログ／CSV数式インジェクション対策／任意でAES-256-GCM暗号化(scrypt鍵導出)
  • 最終レビュー：弁護士＋（必要に応じ）個人情報保護委員会への事前確認運用
